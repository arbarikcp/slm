# Module 5.7 — Speculative Decoding

> **Goal:** Cut decoder latency 2–3× without changing model weights or output distribution — benchmark DeskMate's decoder with speculative decoding vs standard decoding on tokens/sec, acceptance rate, and quality.

---

## The Problem with Autoregressive Decoding

Standard transformer decoding is **inherently sequential**: each token is generated one at a time, requiring one full forward pass of the target model per token. For a 1.5B parameter model on a T4 GPU:

- Each forward pass: ~35–50 ms
- Generating a 100-token reply: ~3.5–5 seconds
- The GPU is mostly **idle** during these passes — the forward pass is memory-bandwidth-bound, not compute-bound, so the GPU's arithmetic units are underutilised

The bottleneck is not the GPU's flops — it is the number of sequential steps.

---

## The Speculative Decoding Insight

**Key observation:** a target model's forward pass with a batch of `N` positions takes barely longer than with 1 position — because the attention over `N` positions is computed in parallel.

**Speculative decoding** exploits this:

1. A small, fast **draft model** generates `N` token proposals in `N` sequential steps (cheap, because the draft model is tiny)
2. The large **target model** verifies all `N` proposals in a **single parallel forward pass**
3. Tokens where the draft and target agree are accepted; the first disagreement is resampled from the target's distribution; the sequence continues

Net effect: `N` tokens for approximately the cost of `1` target forward pass + `N` draft forward passes (where draft passes are ~10–50× cheaper than target passes).

---

## The Algorithm

### Setup

- **Target model** `M_p`: the full DeskMate decoder (Qwen2.5-1.5B, distribution `p`)
- **Draft model** `M_q`: a smaller model of the same family (e.g. Qwen2.5-0.5B, distribution `q`)
- **Draft length** `γ`: number of tokens the draft proposes per step (typically 4–8)

### One speculative decoding step

```
Given context tokens x_1, ..., x_t:

1. Draft model generates γ candidate tokens:
   x̃_{t+1}, x̃_{t+2}, ..., x̃_{t+γ}
   (using greedy or temperature sampling from q)

2. Target model runs ONE forward pass over:
   x_1, ..., x_t, x̃_{t+1}, ..., x̃_{t+γ}
   producing probabilities p(· | x_1..x_t), p(· | x_1..x_{t+1}), ...

3. For each position i = 1..γ, accept x̃_{t+i} with probability:
   α_i = min(1, p(x̃_{t+i} | context) / q(x̃_{t+i} | context))

4. If all γ accepted: output x̃_{t+1..t+γ}, sample one more from target,
   advance context by γ+1

   If position k is first rejection (k ≤ γ):
   - Keep x̃_{t+1..t+k-1}
   - Resample x_{t+k} from corrected distribution:
     p'(x) = max(0, p(x) - q(x)) / Z  (Z = normalisation)
   - Advance context by k
```

### Why output is statistically identical to target-only sampling

The acceptance criterion `min(1, p/q)` is the **rejection sampling** rule. It guarantees that the marginal distribution of each accepted token is exactly `p` — the target distribution. The resampled token at the rejection point is also drawn from a corrected distribution that exactly matches `p`. Because every token in the output is drawn from `p` (either via acceptance or via correction), the output sequence is **statistically indistinguishable** from standard target-only sampling. This is not an approximation.

This is the key property: speculative decoding is a **lossless** speedup. Quality metrics (ROUGE-L, LLM judge, etc.) should be identical to the baseline. If they differ, the implementation is wrong.

---

## Acceptance Rate α

**α** is the fraction of draft tokens that are accepted by the target:

```
α = E[ number of accepted tokens per γ-token batch ] / γ
```

α depends on how well the draft model's distribution `q` matches the target's distribution `p`:

| Scenario | Typical α |
|---|---|
| Draft and target are same family, same domain | 0.7–0.85 |
| Draft is from a different model family | 0.5–0.7 |
| Draft is fine-tuned on DeskMate data, target is also fine-tuned | 0.75–0.90 |
| Greedy target (temperature=0), greedy draft | Highest α |
| High temperature sampling (temperature=1.0) | Lower α |

### Why domain matters

The draft model's probability `q(x)` must be close to `p(x)` for high acceptance. If you use a generic Qwen2.5-0.5B draft model with a DeskMate-fine-tuned Qwen2.5-1.5B target, `q` was not trained on support ticket language — so the draft often proposes different tokens than the fine-tuned target prefers. Solution: fine-tune the draft model on the same DeskMate dataset.

---

## Speedup Formula

The expected speedup factor relative to single-token decoding:

```
Speedup ≈ (1 + γ·α) / (1 + γ·C_draft/C_target)
```

Where:
- `γ` = draft length
- `α` = acceptance rate
- `C_draft` = cost (time) of one draft forward pass
- `C_target` = cost (time) of one target forward pass

For DeskMate (Qwen2.5-1.5B target, Qwen2.5-0.5B draft, γ=5, α=0.75, C_draft/C_target≈0.15):

```
Speedup ≈ (1 + 5×0.75) / (1 + 5×0.15)
         = 4.75 / 1.75
         ≈ 2.7×
```

Practical speedups are 2–3× on GPU and somewhat lower on CPU (the parallel verification step benefits most from GPU parallelism).

## Mermaid: Speculative Decoding Flow

```mermaid
sequenceDiagram
    participant D as Draft model (small, fast)
    participant T as Target model (large)
    D->>D: generate N draft tokens in N sequential steps
    D->>T: send (context + N draft tokens) in one batch
    T->>T: verify all N positions in one parallel forward pass
    T-->>D: accept/reject each position
    Note over T: accepted tokens kept; first rejected position resampled from target
```

## Implementation with HuggingFace

HuggingFace `transformers` supports speculative decoding natively via `assistant_model`:

```python
from transformers import AutoModelForCausalLM, AutoTokenizer

target = AutoModelForCausalLM.from_pretrained(
    "Qwen/Qwen2.5-1.5B-Instruct",
    torch_dtype=torch.float16,
    device_map="auto",
)
draft = AutoModelForCausalLM.from_pretrained(
    "Qwen/Qwen2.5-0.5B-Instruct",
    torch_dtype=torch.float16,
    device_map="auto",
)
tokenizer = AutoTokenizer.from_pretrained("Qwen/Qwen2.5-1.5B-Instruct")

inputs = tokenizer(prompt, return_tensors="pt").to("cuda")

# Standard decoding
out_std = target.generate(**inputs, max_new_tokens=100, do_sample=False)

# Speculative decoding (assistant_model = draft)
out_spec = target.generate(
    **inputs,
    max_new_tokens=100,
    do_sample=False,
    assistant_model=draft,    # triggers speculative decoding
    # num_assistant_tokens=5, # γ — draft length (default=5)
)
```

The output tokens are **identical** in greedy decoding (`do_sample=False`). The tokens/sec is 2–3× higher.

### Measuring acceptance rate

```python
# HuggingFace stores accepted draft tokens in the generation output
# Use a custom generate loop to count:
from transformers import GenerationConfig

# Or use the assistant_early_exit_threshold parameter to log acceptance
# Alternatively, count matching positions:
def estimate_alpha(std_ids, spec_ids):
    # How many tokens were generated "for free" (in speculative steps)
    n = min(len(std_ids), len(spec_ids))
    matches = sum(a == b for a, b in zip(std_ids[:n], spec_ids[:n]))
    return matches / n
```

## When Speculative Decoding Helps Most

| Condition | Impact on speedup |
|---|---|
| Long outputs (> 50 tokens) | Higher benefit — more speculation steps amortise the draft overhead |
| Low temperature / greedy | Higher α → bigger speedup |
| Draft same family as target | Higher α |
| GPU inference (not CPU) | Parallel verification is GPU-native; CPU benefit is smaller |
| Memory bandwidth-bound target | Biggest win — speculation reduces sequential passes |

For DeskMate: support ticket replies are typically 50–150 tokens, greedy decoding is preferred (deterministic), and inference is on GPU → all three conditions favour speculative decoding. **Expected speedup: 2–2.5× on T4.**

## Checkpoint

> *Why does correct speculative decoding produce outputs that are statistically identical to standard decoding from the target model?*

The acceptance criterion — accept draft token `x̃` with probability `min(1, p(x̃)/q(x̃))` — is exactly the rejection sampling rule for drawing from distribution `p` using proposal `q`. Each accepted token's marginal distribution is `p`, not `q`. When a draft token is rejected at position `k`, the replacement token is sampled from a corrected distribution `p'(x) = max(0, p(x) − q(x)) / Z`, which is also derived to have marginal `p` (it is the "residual" of `p` not covered by `q`). Since every token in the output — whether accepted or resampled — is drawn from `p`, the output sequence has the same distribution as if the target model had generated each token independently. Speculative decoding is therefore a provably lossless optimisation, not an approximation.
