## Why Read-Level?

The three techniques here — SmoothQuant, FlexGen, and BitNet — solve real problems, but each requires specialised infrastructure that is not yet standard in the HuggingFace/bitsandbytes stack DeskMate runs on. You need to know what they solve so you can:

- Evaluate claims like "our model runs on 4 GB RAM in 2-bit" without being fooled
- Spot when DeskMate's scale or hardware would actually benefit
- Know which to reach for if DeskMate grows to 7B+ parameters

## SmoothQuant

### The problem it solves

Standard int8 quantisation (LLM.int8(), Module 5.2) handles **weight outliers** — large values in a small number of weight channels — by routing those channels through fp16. But it does not fix **activation outliers**.

In practice, transformer activations are harder to quantise than weights:
- Activations vary per input (they depend on the token, not just the model)
- Outlier activations are not consistent across tokens — they appear in different channels for different inputs
- LLM.int8()'s per-token outlier detection handles this, but at the cost of a mixed-precision MatMul

SmoothQuant asks: *can we migrate the quantisation difficulty from activations to weights, which are easier to quantise?*

### The key insight

For a linear layer `Y = X · W`, we can introduce a per-channel scale vector **s** without changing the output:

```
Y = (X / s) · (W · diag(s))
  = X_smooth · W_smooth
```

If we choose **s** so that `max(|X_j|) / s_j ≈ max(|W_j|) * s_j` for each channel `j`, we balance the difficulty between activations and weights. Both become easier to quantise to int8, enabling a **pure int8 MatMul** (no mixed-precision fallback).

### Why this matters

Pure int8 MatMul is supported directly in hardware (NVIDIA Tensor Cores, Intel VNNI). Mixed-precision MatMul (LLM.int8()) needs two separate compute paths. SmoothQuant enables:
- True W8A8 (weights int8, activations int8) quantisation
- 1.5–2× faster MatMul than fp16 on supported hardware
- Near-zero quality loss at int8 (perplexity within 0.1 of fp16)

### When to consider for DeskMate

SmoothQuant pays off at 7B+ parameters where activation outliers cause meaningful quality loss in W8A8. For DeskMate at 1.5B, LLM.int8() already preserves quality well enough.

**Paper:** SmoothQuant: Accurate and Efficient Post-Training Quantization for Large Language Models (Xiao et al., 2022)

## FlexGen

### The problem it solves

A 30B or 70B model does not fit in any single GPU's VRAM. Standard inference requires either:
- Multiple GPUs (expensive)
- Model parallelism with NVLink (complex, requires matched hardware)

FlexGen asks: *can a single commodity GPU run inference on models that are 10× too large for its VRAM, by offloading to CPU RAM and SSD?*

### The approach

FlexGen separates the model's memory into three tiers and schedules data movement to overlap compute with I/O:

```
GPU VRAM (fast, small)  →  active layer weights + KV cache
CPU RAM  (medium, large) →  next-layer weights (prefetched)
SSD      (slow, huge)    →  distant layers + overflow KV cache
```

Key ideas:

1. **Computation scheduling:** process one layer at a time. While the GPU runs layer `i`, the CPU prefetches layer `i+1` from RAM (or SSD) into a CPU buffer, then to GPU — hiding transfer latency behind compute.

2. **Batch-first generation:** FlexGen maximises throughput (not latency) by processing large batches. A single A10 GPU can generate outputs for a batch of 256 sequences through a 30B model — slowly (~1 token/sec/seq), but without needing a second GPU.

3. **KV cache offloading:** the KV cache (which grows with sequence length) is also tiered. Short sequences stay in VRAM; long sequences spill to CPU RAM.

### Trade-off

| Property | Standard GPU inference | FlexGen |
|---|---|---|
| Latency per token | Low (~50 ms) | High (~1–10 sec) |
| Throughput (tokens/sec total) | Moderate | High (large batch) |
| Hardware required | Multiple GPUs for large models | 1 GPU + CPU RAM + SSD |
| Use case | Interactive serving | Offline batch processing |

FlexGen is not for real-time support chat. It is for **offline batch jobs** — generating answers for 10 000 queued tickets overnight on one GPU.

### When to consider for DeskMate

If DeskMate ever needs to run a 30B+ model for a high-stakes reasoning task (e.g., complex legal policy interpretation) in a batch overnight job on a single GPU node — FlexGen is the right tool.

**Paper:** FlexGen: High-Throughput Generative Inference of Large Language Models with a Single GPU (Sheng et al., 2023)

## BitNet

### The problem it solves

All quantisation schemes so far still use multi-bit integers (int8, int4). BitNet asks: *what is the minimum bit-width that preserves language model quality — and can we train a model specifically for that precision from scratch?*

The answer explored in BitNet: **1-bit weights** (values in {−1, +1}) or **1.58-bit weights** (values in {−1, 0, +1}).

### How it works

Standard quantisation is **post-training**: train at fp16, then quantise. BitNet is **quantisation-aware training** at the extreme:

- Weights are constrained to ternary values {−1, 0, +1} during training via a `sign()` operation with a straight-through estimator for gradients
- Activations are quantised to int8 per token
- MatMul becomes a series of **additions and subtractions** (multiplying by ±1 or 0 = no multiply needed)

### BitNet 1.58b results

Microsoft Research's BitNet b1.58 paper (2024) showed:
- A 3B ternary model matches a 3B fp16 model on perplexity — when trained from scratch in ternary
- Inference uses integer addition/subtraction only — no floating-point multiplications in the weight layers
- Memory: 3B model ≈ 0.7 GB in 1.58-bit vs 6 GB in fp16

### What BitNet does NOT fix

- **Requires training from scratch.** You cannot post-train-quantise an existing fp16 model to 1-bit; the quality collapse is catastrophic. A ternary model must be initialised and trained with quantisation-aware objectives from the start.
- **Ecosystem immaturity.** No production-ready runtime as of mid-2025. llama.cpp has experimental support; HuggingFace integration is incomplete.
- **Not a drop-in replacement for DeskMate's Qwen base.** DeskMate's Qwen2.5-1.5B was trained in bf16. Converting it post-hoc to BitNet is not the same technique and would degrade quality severely.

### When to consider for DeskMate

Only if DeskMate migrates to a base model that was **pre-trained as a BitNet model** (e.g., Microsoft releases a BitNet-trained Phi or LLaMA variant at scale). At that point, inference would be dramatically cheaper per token.

**Papers:**
- BitNet: Scaling 1-bit Transformers for Large Language Models (Wang et al., 2023)
- The Era of 1-bit LLMs: All Large Language Models are in 1.58 Bits (Ma et al., 2024)

|---|---|---|---|

## Checkpoint

> *One sentence each on what SmoothQuant, FlexGen, and BitNet solve.*

**SmoothQuant:** Migrates quantisation difficulty from hard-to-quantise activations to easy-to-quantise weights via per-channel scaling, enabling a pure W8A8 int8 MatMul with near-zero quality loss.

**FlexGen:** Enables single-GPU inference on models 10× larger than VRAM by tiering weights and KV cache across GPU/CPU/SSD and overlapping data prefetch with computation — trading latency for throughput in offline batch settings.

**BitNet:** Trains language models from scratch with ternary weights (−1, 0, +1) so that inference reduces to integer additions rather than floating-point multiplications, achieving 8× memory reduction vs fp16 at matched quality — but requires training from scratch, not post-training conversion.
