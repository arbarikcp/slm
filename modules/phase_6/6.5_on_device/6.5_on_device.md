# Module 6.5 — On-Device / Mobile (Overview)

> **Goal:** Understand the edge-deployment path — what MLC LLM does, how a model gets to Android, and what fundamentally changes when your inference server is a phone.

---

## What Changes When Your "Server" Is a Phone

Mobile inference is not just "smaller GPU." The entire systems contract changes:

| Dimension | Server (GPU) | Phone (edge) |
|---|---|---|
| Memory | 16–80 GB VRAM + RAM | 4–12 GB shared RAM (no dedicated VRAM) |
| Thermal throttling | Sustained full power | Throttles after ~30–60 s under load |
| Battery | Mains power, unlimited | Battery-constrained; LLM inference drains fast |
| Connectivity | Always online | May be offline; network unreliable |
| Latency SLA | User-facing web app | Immediate UI response expected |
| Model updates | Redeploy container | App update through Play Store / App Store |
| Hardware diversity | 1–2 GPU types per deployment | Hundreds of Android SoC variants |

The phone is not a slow GPU — it is a different class of compute with different constraints.

---

## MLC LLM

**MLC LLM** (Machine Learning Compilation for LLMs) is the primary tool for deploying language models on mobile and edge devices:

- **What it is:** a compilation framework that takes a model (Llama, Mistral, Qwen, etc.) and compiles it to run natively on a specific device's hardware — Android GPU (OpenCL/Vulkan), iOS GPU (Metal), WebGPU, or ARM CPU
- **Output:** a compiled model artefact (`.mlc` format) + a runtime library (`libmlc_llm.so` on Android)
- **Languages:** Python (development), C++ and Java/Kotlin (Android app), Swift (iOS)

### Why compilation matters for mobile

Mobile SoCs (Qualcomm Snapdragon, Samsung Exynos, MediaTek Dimensity) have heterogeneous compute units — CPU, GPU, NPU (Neural Processing Unit). Each unit has different instruction sets and memory layouts. A generic ONNX or PyTorch model cannot use the NPU at all.

MLC LLM's compiler emits **device-specific kernels** that target the actual hardware, achieving 3–10× higher throughput than a generic runtime on the same chip.

## Compilation Pipeline

```bash
# 1. Convert HuggingFace model → MLC format
mlc_chat convert_weight Qwen/Qwen2.5-1.5B-Instruct \
    --quantization q4f16_1 \
    --output models/deskmate-mlc/

# 2. Compile for Android (OpenCL backend, Qualcomm GPU)
mlc_chat compile models/deskmate-mlc/params/ \
    --device android \
    --quantization q4f16_1 \
    --output models/deskmate-android/

# 3. Package into an Android app
# See MLC LLM's Android template project:
# https://github.com/mlc-ai/mlc-llm/tree/main/android
```

The compiled artefact is ~1 GB for a 1.5B Q4 model — small enough to ship in an APK (with split APK delivery) or download on first launch.

## Android Integration

MLC LLM provides a Java/Kotlin API for Android apps:

```kotlin
import ai.mlc.mlcllm.ChatModule

val chat = ChatModule()
chat.reload("deskmate-android", "q4f16_1")

val response = chat.generate(
    "Ticket: I was charged twice last month.",
    streamCallback = { token -> textView.append(token) }
)
```

The model runs **entirely on-device** — no network call, no API key, no server. The app works offline.

## Quantisation for Mobile

Standard 4-bit quantisation (Module 5.3) must be adapted for mobile:

| Format | Mobile-compatible? | Notes |
|---|---|---|
| GGUF Q4_K_M | Only via llama.cpp CPU path | Not GPU-accelerated on Android |
| MLC q4f16_1 | ✅ Android/iOS GPU via OpenCL/Metal | Best mobile throughput |
| MLC q4f32_1 | ✅ | Higher quality, larger file |
| MLC q0f16 | ✅ (fp16) | Best quality, ~3 GB, only fits flagship phones |

**q4f16_1** is the standard mobile target: weights in int4, activations in fp16, 1 group per 32 weights. On a Snapdragon 8 Gen 3, a 1.5B model at q4f16_1 achieves ~25–40 tokens/sec.

## Constraints of Mobile Inference

### Memory

A phone's RAM is **shared** between the OS, apps, and the model. On an 8 GB Android device:
- OS + baseline apps: ~3–4 GB
- Available for model + KV cache: ~2–4 GB
- DeskMate (1.5B, q4f16_1): ~1.2 GB weights + ~0.3 GB KV cache = ~1.5 GB ✅

A 7B model at q4f16_1 (~4.5 GB) would OOM most phones. Model size is a hard constraint.

### Thermal throttling

A phone's SoC cannot sustain full-speed inference indefinitely. After 30–60 seconds of continuous load, the thermal management unit reduces clock speeds — throughput can drop to 30–50% of peak. For support use cases (one reply per query, not continuous streaming), this is usually not a problem.

### Update path

Updating the model requires an app update through the app store — with review delays, staged rollouts, and users on old versions. Server-side deployment can push model updates instantly. This is a significant operational difference: on-device models cannot be hotfixed.

## WebGPU: Browser-Based Inference

MLC LLM also targets WebGPU — the new web standard for GPU compute in browsers:

```javascript
import { ChatModule } from "@mlc-ai/web-llm";

const chat = new ChatModule();
await chat.reload("deskmate-q4f16_1-MLC");
const reply = await chat.generate("Ticket: My CSV export is broken.");
console.log(reply);
```

This runs DeskMate in a browser tab, using the GPU via WebGPU — no server, no Python, no API key. Works on Chrome/Edge with a modern GPU. A 1.5B model loads in ~5 seconds on a MacBook and generates at ~15 tok/sec.

## DeskMate Mobile Architecture

```
Support agent's phone (offline capable)
  │
  ├─ MLC LLM runtime (compiled q4f16_1 model, ~1.5 GB)
  │    inference: ~25-40 tok/sec on Snapdragon 8 Gen 3
  │
  ├─ FAISS index (local copy, ~50 MB for DeskMate FAQ)
  │    updates: sync on WiFi when available
  │
  └─ DeBERTa encoder (ONNX, ~90 MB)
       runs on CPU via ONNX Runtime Mobile
```

The full pipeline works offline. When connectivity is available, the FAQ index is synced from a central server.

## Checkpoint

> *What changes when your "server" is a phone?*

Five things fundamentally change:

1. **Shared memory, no VRAM:** the phone has no dedicated GPU memory — model weights, KV cache, OS, and all other apps compete for the same RAM pool. Model size is limited to ~2 GB on most devices.
2. **Thermal throttling:** sustained inference causes the SoC to heat up and reduce clock speeds after 30–60 seconds. Throughput is not stable over time.
3. **Battery constraint:** LLM inference is one of the most power-intensive workloads a phone can run. A 100-token reply at full speed consumes ~100–200 mAh. User experience requires careful power management.
4. **Hardware diversity:** Android runs on hundreds of SoC variants. A model compiled for Snapdragon may perform differently on Exynos or Dimensity. MLC LLM's compiler targets specific hardware, but testing coverage is limited.
5. **Update path:** changing the model requires shipping a new app version through the app store review process — not a server redeploy. Model versioning and rollback are fundamentally different operational problems.
