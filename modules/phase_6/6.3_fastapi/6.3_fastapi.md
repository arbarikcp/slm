## The Full Pipeline as a Service

DeskMate's production request path has three components, each with different latency characteristics:

```
Request
  │
  ├─► Encoder (DeBERTa, Module 2.5)        ~2–22 ms   (ONNX + I/O binding)
  │     classify intent + extract fields
  │
  ├─► Retriever (FAISS + BM25 + reranker, Module 4.3)   ~30–80 ms
  │     fetch top-3 relevant chunks
  │
  └─► Decoder (Qwen2.5-1.5B via vLLM, Modules 3.4 / 6.2)  ~1–4 s
        generate grounded reply with citations
```

The encoder and retriever are fast enough to run synchronously. The decoder dominates latency — vLLM's continuous batching is the primary tool for managing it under load.

---

## Where Each Component Goes in the Request Path

**Encoder (synchronous, before decoder):**
The encoder runs first to classify intent and extract `{product, issue_type}` fields. This happens before retrieval — the extracted fields are used as metadata filters in the FAISS + Chroma query. It cannot be deferred.

**Retriever (synchronous, before decoder):**
The retriever uses the ticket text + extracted fields to fetch the top-3 chunks. These chunks are assembled into the RAG prompt (Module 4.4) before the decoder call. Cannot be deferred.

**Decoder (the bottleneck):**
The decoder is the only component that benefits from batching and continuous scheduling. All decoder calls go through vLLM. The FastAPI service calls vLLM's API and streams the response back to the client.

## API Design

### `/triage` — encoder only

Fast endpoint: classify the ticket without generating a full reply. Useful for routing tickets to human agents or queues.

```
POST /triage
{
  "ticket": "I was charged twice for my subscription."
}

→ 200 OK
{
  "intent": "billing_dispute",
  "product": "DeskMate Pro",
  "confidence": 0.94,
  "latency_ms": 18
}
```

### `/answer` — full pipeline

Slow endpoint: encode → retrieve → generate. Returns a grounded reply with citations.

```
POST /answer
{
  "ticket": "I was charged twice for my subscription.",
  "stream": false
}

→ 200 OK
{
  "reply": "If you were charged twice, contact support with your invoice numbers. [Source 1]",
  "citations": [1],
  "intent": "billing_dispute",
  "faithfulness_score": 0.91,
  "latency_ms": 1840
}
```

### `/answer` with streaming

For interactive UIs, the decoder streams tokens as they are generated:

```
POST /answer
{"ticket": "...", "stream": true}

→ 200 OK  Content-Type: text/event-stream
data: {"token": "If"}
data: {"token": " you"}
data: {"token": " were"}
...
data: [DONE]
```

## FastAPI Implementation

```python
from fastapi import FastAPI
from pydantic import BaseModel
from fastapi.responses import StreamingResponse
import httpx, time, json

app = FastAPI(title="DeskMate API")

VLLM_URL = "http://localhost:8000/v1/chat/completions"
SYSTEM_MSG = "You are DeskMate. Answer using ONLY the provided context. Cite with [Source N]."


class TicketRequest(BaseModel):
    ticket: str
    stream: bool = False


@app.post("/triage")
async def triage(req: TicketRequest):
    t0 = time.perf_counter()
    intent, product, confidence = encoder_predict(req.ticket)
    return {
        "intent": intent,
        "product": product,
        "confidence": confidence,
        "latency_ms": round((time.perf_counter() - t0) * 1000),
    }


@app.post("/answer")
async def answer(req: TicketRequest):
    t0 = time.perf_counter()
    intent, product, _ = encoder_predict(req.ticket)
    chunks = full_retrieve(req.ticket, fields={"product": product})
    prompt = build_rag_prompt(req.ticket, chunks)

    if req.stream:
        return StreamingResponse(
            _stream_vllm(prompt), media_type="text/event-stream")

    async with httpx.AsyncClient(timeout=30) as client:
        r = await client.post(VLLM_URL, json={
            "model": "Qwen/Qwen2.5-1.5B-Instruct",
            "messages": prompt,
            "max_tokens": 200,
            "temperature": 0.0,
        })
    reply = r.json()["choices"][0]["message"]["content"]
    citations = extract_citations(reply, len(chunks))
    faith = faithfulness_score(reply, chunks)
    return {
        "reply": reply,
        "citations": citations,
        "intent": intent,
        "faithfulness_score": faith,
        "latency_ms": round((time.perf_counter() - t0) * 1000),
    }


async def _stream_vllm(prompt):
    async with httpx.AsyncClient(timeout=60) as client:
        async with client.stream("POST", VLLM_URL, json={
            "model": "Qwen/Qwen2.5-1.5B-Instruct",
            "messages": prompt,
            "max_tokens": 200,
            "temperature": 0.0,
            "stream": True,
        }) as response:
            async for line in response.aiter_lines():
                if line.startswith("data: ") and line != "data: [DONE]":
                    chunk = json.loads(line[6:])
                    delta = chunk["choices"][0]["delta"].get("content", "")
                    if delta:
                        yield f"data: {json.dumps({'token': delta})}\n\n"
            yield "data: [DONE]\n\n"
```

## Benchmarking Variants

You have three serving variants to compare:

| Variant | Stack | Expected p50 |
|---|---|---|
| **A** | HuggingFace generate, fp16, batch=1 | ~3.5 s |
| **B** | vLLM, fp16, continuous batching | ~1.4 s |
| **C** | vLLM, int4 (GPTQ), continuous batching | ~0.9 s |

Benchmark script:

```python
import asyncio, httpx, time, statistics

API_URL = "http://localhost:8001"  # your FastAPI service

TICKETS = [
    "I was charged twice for my subscription.",
    "How do I reset my 2FA device?",
    "The CSV export is broken.",
    "Can I get a refund after 45 days?",
    "My iOS app crashes on iPhone 14.",
] * 10  # 50 total requests

async def single_request(client, ticket):
    t0 = time.perf_counter()
    r = await client.post(f"{API_URL}/answer", json={"ticket": ticket}, timeout=30)
    return (time.perf_counter() - t0) * 1000  # ms

async def load_test(concurrency=4):
    sem = asyncio.Semaphore(concurrency)
    async def bounded(client, ticket):
        async with sem:
            return await single_request(client, ticket)
    async with httpx.AsyncClient() as client:
        tasks = [bounded(client, t) for t in TICKETS]
        latencies = await asyncio.gather(*tasks)
    return {
        "p50_ms": round(statistics.median(latencies)),
        "p99_ms": round(statistics.quantiles(latencies, n=100)[98]),
        "throughput_rps": round(len(TICKETS) / (max(latencies)/1000), 2),
    }
```

## Picking and Deploying the Best Performer

Decision framework:

```
If p99 < 2 s at target concurrency:
    → deploy vLLM fp16 (best quality, easiest to debug)
Elif p99 < 2 s only with int4:
    → deploy vLLM GPTQ (quality delta < 0.01 ROUGE-L from Module 5.3)
Else:
    → split: use /triage (fast) for routing, queue /answer for async delivery
```

## Checkpoint

> *Where do you put the encoder vs decoder vs retriever in the request path?*

**Encoder first, synchronously:** The encoder runs before retrieval because its output (`intent`, `product`) is used as a metadata filter in the vector search. It is fast enough (< 25 ms with ONNX) that it never blocks the response.

**Retriever second, synchronously:** The retriever depends on the encoder output and must produce chunks before the decoder can be called. At 30–80 ms it is the second-fastest stage. It too runs synchronously in the request path.

**Decoder last, via vLLM:** The decoder is the bottleneck. It runs last because it needs the retrieved chunks to build the RAG prompt. Routing it through vLLM instead of calling `model.generate()` directly gives continuous batching, paged attention, and OpenAI-compatible streaming — all for free.

The encoder and retriever are **pre-processing** (fast, synchronous); the decoder is **the service call** (slow, needs the engine).
