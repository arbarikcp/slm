## Why Local Deployment Is a Feature

For many enterprise support teams, local deployment is not a fallback — it is a **requirement**:

1. **Data privacy:** support tickets often contain PII (customer names, account numbers, transaction IDs). Sending them to a cloud API means the data leaves your security boundary. Local inference keeps it on-premise.
2. **Regulatory compliance:** GDPR, HIPAA, SOC 2, and industry-specific regulations may prohibit sending sensitive ticket content to third-party APIs. Local inference is the only compliant option.
3. **Offline availability:** field service teams, air-gapped environments, or locations with poor connectivity need the model to work without internet access.
4. **Latency and cost certainty:** no per-token cloud billing, no API rate limits, no network latency variance.

Local deployment is not a convenience — it is a product feature that unlocks regulated industries and high-security customers.

## Ollama

Ollama wraps `llama.cpp` with a model management layer and a REST API:

- **`ollama pull`** — downloads a model from the Ollama registry
- **`ollama create`** — builds a custom model from a Modelfile (your GGUF + instructions)
- **`ollama run`** — interactive chat in the terminal
- **REST API** on `localhost:11434` — OpenAI-compatible `/api/chat` and `/api/generate`
- **Model library** — hundreds of models, or bring your own GGUF

## Importing DeskMate into Ollama

The output of Module 5.3 is a GGUF file: `models/deskmate-Q4_K_M.gguf`. Import it via a Modelfile:

### Modelfile

```dockerfile
# Modelfile for DeskMate
FROM ./models/deskmate-Q4_K_M.gguf

# System prompt — DeskMate's identity and grounding instruction
SYSTEM """
You are DeskMate, a concise support assistant.
Answer the ticket using ONLY the provided context passages.
Cite each claim with [Source N] where N is the passage number.
If the context does not contain the answer, respond with:
'I don't have that information — please contact support@deskmate.com.'
"""

# Generation parameters
PARAMETER temperature 0.0
PARAMETER num_predict 200
PARAMETER num_ctx 2048
PARAMETER stop "<|im_end|>"
PARAMETER stop "<|endoftext|>"
```

### Build and run

```bash
# Build the Ollama model from the Modelfile
ollama create deskmate -f Modelfile

# Verify it appears in the local registry
ollama list

# Interactive chat (terminal)
ollama run deskmate "Ticket: I was charged twice last month."

# API call (same interface as vLLM in Module 6.2)
curl http://localhost:11434/api/chat -d '{
  "model": "deskmate",
  "messages": [{"role": "user", "content": "Ticket: I was charged twice."}],
  "stream": false
}'
```

## Calling Ollama from Python

Ollama exposes an OpenAI-compatible endpoint at `/v1/` (Ollama ≥ 0.1.24):

```python
from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:11434/v1",
    api_key="ollama",  # required by the client, not validated
)

response = client.chat.completions.create(
    model="deskmate",
    messages=[
        {"role": "user", "content": "Ticket: I was charged twice last month."},
    ],
    max_tokens=200,
    temperature=0.0,
)
print(response.choices[0].message.content)
```

Or use the `ollama` Python package directly:

```python
import ollama

response = ollama.chat(
    model="deskmate",
    messages=[{"role": "user", "content": "Ticket: I was charged twice."}],
)
print(response["message"]["content"])
```

## Performance: GGUF Q4_K_M on CPU

For DeskMate (Qwen2.5-1.5B, Q4_K_M, 0.95 GB):

| Hardware | Tokens/sec | 100-token reply |
|---|---|---|
| M2 MacBook Pro (10-core CPU) | ~35 tok/s | ~3 s |
| Intel i7-12th gen (8-core) | ~12 tok/s | ~8 s |
| T4 GPU (via llama.cpp CUDA) | ~80 tok/s | ~1.3 s |

CPU inference is slower than GPU vLLM, but for most support ticket use cases (async, not real-time chat), 8–12 seconds per reply is acceptable.

## LM Studio

LM Studio is a desktop GUI alternative to Ollama:

- **Download:** lmstudio.ai
- **Load model:** File → Open → select `deskmate-Q4_K_M.gguf`
- **Chat UI:** interactive conversation in the app
- **Local API server:** start the server in LM Studio → use at `http://localhost:1234/v1/`

LM Studio is useful for **non-technical users** who need to run DeskMate locally without any command-line setup.

### LM Studio Python SDK

```python
from lmstudio import LMStudio

client = LMStudio()  # connects to running LM Studio server
result = client.chat(
    model="deskmate-Q4_K_M",
    messages=[{"role": "user", "content": "Ticket: CSV export is broken."}],
)
print(result.content)
```

## Full Local Pipeline

With Ollama running, the full DeskMate pipeline can run locally:

```python
import ollama, faiss, numpy as np
from sentence_transformers import SentenceTransformer

# Encoder runs locally (no GPU needed for deberta-v3-small at inference)
enc = SentenceTransformer("microsoft/deberta-v3-small")

# Retriever uses the FAISS index built in Module 4.2 (already on disk)
index = faiss.read_index("data/deskmate_faq.index")

# Decoder: Ollama (local GGUF, CPU)
def answer_ticket(ticket):
    q_emb = enc.encode(ticket)
    _, ids = index.search(np.array([q_emb]), k=3)
    chunks = [chunk_records[i] for i in ids[0]]
    context = "\n\n".join(f"[Source {i+1}] {c['text']}" for i, c in enumerate(chunks))
    return ollama.chat(
        model="deskmate",
        messages=[{"role": "user", "content": f"Context:\n{context}\n\nTicket: {ticket}"}],
    )["message"]["content"]

print(answer_ticket("I was charged twice last month."))
```
