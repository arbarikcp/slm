## 1. Colab vs Kaggle — Know What You Are Using

Both are free. Both give you a T4 GPU. They have different strengths and limits.

| | Google Colab (free) | Kaggle (free) |
|---|---|---|
| GPU | T4 (16 GB VRAM) | T4 × 2 or P100 (16 GB) |
| GPU quota | ~4–6 hrs/session; soft weekly limit | 30 GPU-hrs / week (hard) |
| Session lifetime | Up to 12 hrs (idle cutoff ~90 min) | Up to 12 hrs |
| Storage | 100 GB ephemeral (lost on disconnect) | 20 GB persistent in `/kaggle/working` |
| Secrets management | Colab Secrets (left sidebar → 🔑) | Kaggle Secrets (Add-ons → Secrets) |
| Best for | Quick iteration, Colab Pro for longer runs | Longer GPU runs, dual-GPU experiments |
| Upgrade path | Colab Pro ($10/mo), Colab Pro+ ($50/mo) | N/A — fixed quota |

**The critical insight about free-tier storage:** everything in `/content/` on Colab is wiped
when your session ends or disconnects. If you train for 2 hours and don't checkpoint to Google
Drive or the HF Hub, that training is gone. **We fix this with the pattern in Section 4.**

### GPU types you will encounter

| GPU | VRAM | FP16 TFLOPs | Good for |
|---|---|---|---|
| T4 | 16 GB | 65 | Everything in Phases 0–4, encoder fine-tuning, QLoRA 1–3B |
| P100 | 16 GB | 18 (no BF16) | Same tasks, slower; no BF16 support is a limitation |
| A10 | 24 GB | 125 | Faster QLoRA, comfortable 7B inference |
| A100 (40 GB) | 40 GB | 312 | Full fine-tune 1–3B, QLoRA 7B |
| A100 (80 GB) | 80 GB | 312 | Full fine-tune 7B, QLoRA 70B |

For all modules through Phase 4, a free T4 is enough.

## 3. Experiment Tracking with Weights & Biases (W&B)

W&B is free for personal / academic use. It gives you:
- **Live loss curves** — see if your run is healthy without waiting for it to finish.
- **Hyperparameter tables** — compare runs side by side.
- **Artifact versioning** — store datasets, models, and eval results with lineage.
- **System metrics** — GPU utilisation, memory, temperature automatically logged.

### Key W&B concepts

```
Project         ← groups all runs for one product (e.g. "deskmate")
  └── Run       ← one training job (one set of hyperparameters)
        ├── Config   ← hyperparameters logged at start
        ├── Metrics  ← loss, accuracy, etc. logged per step
        └── Artifacts ← model checkpoints, datasets with version history
```

### The minimum you log on every training run

```python
import wandb

wandb.init(
    project="deskmate",
    name="encoder-v1-minilm",       # human-readable run name
    config={                         # all hyperparameters — logged once
        "model": "microsoft/MiniLM-L12-H384-uncased",
        "lr": 2e-5,
        "batch_size": 32,
        "epochs": 3,
    }
)

# inside training loop:
wandb.log({"train/loss": loss.item(), "train/step": step})

# at end:
wandb.finish()
```

W&B then gives you a URL to a live dashboard. Share it; it doesn't expire.

## 4. The "Never Lose a Checkpoint" Pattern

This is the single habit that saves the most time on free-tier compute.

**Why sessions die:**
- Colab idles out after ~90 minutes of no browser activity.
- The free-tier runtime has a 12-hour hard limit.
- Kaggle sessions also have limits.

**The pattern:** save locally *and* push to HF Hub after every epoch. If the session dies,
resume from the last Hub checkpoint — zero work lost.

```python
from huggingface_hub import login
login()  # use token from Colab Secrets / Kaggle Secrets

# After each epoch in your Trainer callback or training loop:
trainer.save_model("models/ckpt-epoch-{epoch}")        # local (ephemeral)
model.push_to_hub("your-username/deskmate-encoder")    # durable — survives disconnect

# To resume after a crash:
model = AutoModel.from_pretrained("your-username/deskmate-encoder")
```

With HF Hub as your durable store, a Colab disconnect is a 2-minute inconvenience, not a
lost training run.

## 5. GPU Hygiene — Making 16 GB Behave Like More

These are the levers you pull when you hit out-of-memory (OOM) errors.

### 5.1 Check GPU health first

```python
import torch

print(f"CUDA available: {torch.cuda.is_available()}")
print(f"GPU name:       {torch.cuda.get_device_name(0)}")
print(f"VRAM total:     {torch.cuda.get_device_properties(0).total_memory / 1e9:.1f} GB")
print(f"VRAM free:      {torch.cuda.memory_reserved(0) / 1e9:.2f} GB reserved")
```

### 5.2 Clear the cache between experiments

```python
import gc
import torch

def clear_gpu():
    gc.collect()
    torch.cuda.empty_cache()
    torch.cuda.synchronize()

# Call this between loading different models in the same session
clear_gpu()
```

### 5.3 Gradient accumulation — simulate large batches

When a batch of 32 doesn't fit in VRAM, use a batch of 4 with 8 accumulation steps.
The gradients accumulate over 8 steps before one optimizer update — mathematically identical
to a batch of 32, at 1/8th the peak memory.

```python
EFFECTIVE_BATCH = 32
MICRO_BATCH     = 4
ACCUM_STEPS     = EFFECTIVE_BATCH // MICRO_BATCH   # = 8

optimizer.zero_grad()
for i, batch in enumerate(dataloader):
    loss = model(batch) / ACCUM_STEPS              # scale loss
    loss.backward()                                 # accumulate gradients
    if (i + 1) % ACCUM_STEPS == 0:
        optimizer.step()                            # one update every 8 steps
        optimizer.zero_grad()
```

In HuggingFace `TrainingArguments`, this is just `gradient_accumulation_steps=8`.

### 5.4 Mixed precision — cut memory and speed up computation

FP16 or BF16 halves the memory of activations and gradients during training.
BF16 is preferred (wider dynamic range, no loss scaling needed) but requires Ampere
GPUs (A10, A100) or newer. T4 only supports FP16.

```python
# In TrainingArguments:
TrainingArguments(
    fp16=True,   # T4 (Turing) — needs GradScaler under the hood
    bf16=False,  # A100/A10 only — simpler, preferred when available
    tf32=True,   # free speedup on Ampere — matmuls in TF32, rest in FP32
)

# Raw PyTorch:
from torch.cuda.amp import autocast, GradScaler

scaler = GradScaler()
with autocast(dtype=torch.float16):
    loss = model(batch)
scaler.scale(loss).backward()
scaler.step(optimizer)
scaler.update()
```

### 5.5 OOM escalation ladder

When you hit OOM, work through this list in order:

```
1. Reduce batch size by half
2. Add gradient accumulation to compensate (double accum_steps)
3. Enable gradient checkpointing  →  saves ~40% memory, ~20% slower
4. Shorten max sequence length
5. Drop to 8-bit loading (bitsandbytes) or 4-bit (QLoRA)
```

Gradient checkpointing trades compute for memory — it recomputes activations on the
backward pass instead of storing them, saving ~40% VRAM at the cost of ~20% slower training.

```python
model.gradient_checkpointing_enable()
```

## 6. Seeding for Reproducibility

Always set seeds at the top of every notebook. Without this, two runs with identical
hyperparameters may produce different results.

```python
import random
import numpy as np
import torch

def set_seed(seed: int = 42):
    random.seed(seed)
    np.random.seed(seed)
    torch.manual_seed(seed)
    torch.cuda.manual_seed_all(seed)

set_seed(42)
```

For full determinism on GPU (at a performance cost), also add:
```python
torch.backends.cudnn.deterministic = True
torch.backends.cudnn.benchmark = False
```

## ✅ Checkpoint Questions

**Q1.** Your Colab session disconnects at epoch 2 of a 5-epoch training run.
What exact steps do you take to resume with zero progress lost?

**Q2.** You are training with `batch_size=32` and hit OOM. You switch to `batch_size=4`
with `gradient_accumulation_steps=8`. What changed mathematically, what stayed the same,
and what is the one thing that is slightly different in practice?

**Q3.** You have a T4 (free Colab) and an A100 (rented). You want to use mixed precision.
Which dtype do you use on each, and why is BF16 preferred on the A100?

---

<details>
<summary>Checkpoint answers — expand only after writing your own</summary>

**A1.** Resume steps:
```
1. Start a new Colab session (or reconnect).
2. Re-run the setup cell (pip install, imports, login tokens).
3. Load the model from HF Hub:
   model = AutoModel.from_pretrained("your-username/deskmate-encoder")
   optimizer.load_state_dict(...)  # if you saved optimizer state too
4. Set the starting epoch to 3 (the last checkpoint was end of epoch 2).
5. Re-run training from that epoch.
```
Total cost: ~5 minutes, zero lost gradient updates.

**A2.** What changed: peak memory per step — only 4 samples are in VRAM at once, so
activations for only 4 samples need to be stored. What stayed the same: the effective
gradient update is computed over 32 samples (accumulated over 8 steps before `optimizer.step()`),
so the parameter update direction and scale are equivalent to a batch of 32.
What is slightly different: when using batch norm layers, statistics are computed over 4
samples per step, not 32 — this can slightly affect training dynamics. For our use cases
(transformer models with LayerNorm, not BatchNorm), this is not an issue.

**A3.** T4 → FP16 (`fp16=True`). T4 is a Turing-architecture GPU; it supports FP16 tensor
cores but not BF16. BF16 is preferred on the A100 (Ampere) because BF16 has the same
exponent range as FP32 (8-bit exponent vs FP16's 5-bit), so gradient values that would
underflow to zero in FP16 are represented correctly in BF16. This means you do not need a
`GradScaler` (loss scaling) with BF16, simplifying the training code and removing one
source of numerical instability.

</details>
