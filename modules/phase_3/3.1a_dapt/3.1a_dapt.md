## What Is DAPT?

Domain-Adaptive Pretraining (DAPT) is a continued pretraining step inserted between the off-the-shelf base model and supervised fine-tuning (SFT):

```
Pretrained base  →  [DAPT on domain corpus]  →  SFT on labeled pairs  →  Production model
```

It runs standard next-token prediction (causal language modelling) on a large corpus of *unlabeled* domain text — support tickets, product documentation, FAQ pages, changelog entries — using the same objective the model was originally pretrained with.

**Crucially: no loss masking.** Every token is a prediction target (unlike SFT where prompt tokens are masked). The DataCollator is `DataCollatorForLanguageModeling(mlm=False)`.

```python
from trl import SFTTrainer
from transformers import DataCollatorForLanguageModeling

trainer = SFTTrainer(
    model           = model,
    train_dataset   = domain_corpus_ds,   # raw text, no labels
    peft_config     = lora_cfg,
    data_collator   = DataCollatorForLanguageModeling(tokenizer, mlm=False),
    args            = cfg,
)
```

## What DAPT Changes

The base model (Qwen2.5-1.5B) was pretrained on general web text. Support-desk language has specific vocabulary, tone, and structure it may not have seen in large quantities:

- Product names: `DeskMate`, `AgentCore`, `FlowPilot`
- Domain jargon: `ticket escalation`, `SLA breach`, `customer churn`
- Error codes, version strings, technical terminology
- Short imperative sentences: "Reset your password from Settings > Account"

DAPT shifts the model's language statistics toward this distribution. After DAPT, the model generates fluent support-desk text more naturally, and SFT requires fewer examples to converge to the target format.

**What it does not change:** the model's world knowledge or reasoning ability. DAPT is purely a language statistics shift.

## When DAPT Is Worth Running

Apply all three criteria before committing compute:

| Criterion | Threshold | DeskMate assessment |
|---|---|---|
| **Domain divergence** | High divergence: domain text looks very unlike general web | Low-to-moderate. Support tickets use common English. Qwen2.5 has seen similar text. |
| **Domain corpus size** | ≥ 50M tokens for meaningful effect | Borderline. Bitext + banking77 + synthetic ≈ 5–10M tokens — below the typical threshold. |
| **SFT gap** | SFT on base alone fails to meet the quality target on gold set | Unknown until Module 3.4 runs. |

**Decision rule:** run DAPT only if SFT on the base model (Module 3.4) fails to meet the gold-set quality target *and* you can source ≥ 50M domain tokens.

---

## Key DAPT Hyperparameters

| Parameter | Typical value | Reasoning |
|---|---|---|
| Learning rate | `1e-5` | Much lower than original pretraining (~1e-3) to avoid catastrophic forgetting |
| Steps | 1k–10k | Enough to shift statistics; too many → overfit to domain vocab, forget general knowledge |
| LoRA vs full weights | LoRA (`r=16`) | Same memory advantage as SFT; full-weight DAPT risks catastrophic forgetting |
| Sequence length | 1024–2048 | Longer context helps the model learn document-level coherence |

## Evaluation: Perplexity on Held-Out Domain Text

DAPT is measured by **perplexity** on a held-out sample of domain text (never seen during DAPT training):

```python
import torch, math
model.eval()
total_loss, n_tokens = 0.0, 0
with torch.no_grad():
    for batch in eval_loader:
        out = model(**batch)
        total_loss += out.loss.item() * batch["input_ids"].numel()
        n_tokens   += batch["input_ids"].numel()
perplexity = math.exp(total_loss / n_tokens)
print(f"Perplexity: {perplexity:.2f}")
```

Compare base model perplexity vs DAPT checkpoint perplexity. A meaningful reduction (>10%) indicates the domain shift worked.

**Important:** lower perplexity on domain text does not guarantee better SFT quality. Always re-evaluate on the gold set after DAPT + SFT to confirm end-to-end improvement.

---

## Checkpoint (For Future Reference)

> *If DAPT lowers perplexity on domain text but SFT quality on the gold set doesn't improve, what does that tell you?*

It tells you the domain shift was real (the model became a better predictor of domain language) but not useful for the downstream task. Possible reasons: the SFT data is the bottleneck (not enough examples, or low quality); the task relies on reasoning and world knowledge that DAPT doesn't improve; or the base model already handles the domain well enough that language-statistics shift provides no marginal benefit. In this case, invest in better SFT data (Module 3.2) rather than more DAPT.
